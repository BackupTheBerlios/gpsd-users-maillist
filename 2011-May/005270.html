On Tue, 31 May 2011 13:25:56 -0400
Eric Raymond <esr at thyrsus.com> wrote:

> > I was suggesting an approach which would have used one single
> > /etc/gpsd.conf instead of the four above.  
> 
> Bad, *bad* idea.  It would involve our C code with the details of the
> Linux /sys interface, which is (a) complex, (b) poorly documented, and
> (c) a moving target.  The hotplug scripts loosen the coupling
> somewhat, reducing total maintainance burden and preventing our core
> code from getting destabilized every time /sys changes.

Also, such code would not be portable to non-Linux OSs. Again, our
primary customers are embedded developers, where hot plugging is usually
an unnecessary complication. Even the notion that gpsd should read a
file has implications for embedded systems that designers might prefer
to avoid.

Also, this would violate the Unix ideal that each program should do one
thing, and only one thing, and do it very very well. gpsd as it stands
complies with that ideal. gpsd should leave hot plugging to the hot
plug code (if any), display to the display code (if any), etc.
