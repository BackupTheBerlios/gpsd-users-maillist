Yo Eric!

On Thu, 31 Mar 2005, Eric S. Raymond wrote:

> Gary E. Miller <gem at rellim.com>:
> > Once again, if the compiler observes pragma pack then this does not
> > apply.  Can someone prove the Zaurus CC is so brain dead it does not
> > have a pragma pack?  Can someone find ANY modern compiler that does not
> > observer pragma pack in some form?
>
> Well, first, I thought we already had a bug report indicating that zgcc
> chokes on #pragma pack.

No.  We have a report the zgcc issues a warning on a cast.

"warning: cast increases required alignment of target type"

Nothing in there leads me to believe the inside of the structure matters.
Only that unsigned char and packet_t may have their base aligned
differently.

I'll bet I can fix that with a slightly different cast or a union.

OTOH, unless the Zaurus has a linux kernel with the garmin driver the
best thing is to fix configure to not compile any of the code in
garmin.c

I can go either way, but unless 2.20 is out I would rather wait though
the "code freeze".

> And second -- even if all compilers support some equivalent of #pragma
> pack, that doesn't mean we're home free.  At least two things could
> still go wrong:
>
> 1. We might not know what the equivalent command-line option or pragma
> or other black magic is, or just not have applied it.  The only way we
> could possibly tell that it *needs* to be applied is when some poor
> user gets back garbage data.

True, then we fix it the way configure fixes up all the other little
nit syntax changes between compiler version.

> 2. We don't control the optimization switches in the target
> environment.  What if one of them turns off #pragma pack, for example
> to do access-time optimization?  The C standard permits this; remember
> that pragmas are advisory only.

Yes, the C standard allows ignoring pragmas.  Can you give me one example
of a modern cc that ignores pragma pack?

> The bottom line is that relying on #pragma pack is bad practice.  Yes,
> I know this is a pain in the butt.  I'd rather use your pack-and-cast method
> myself, it does make the code easier to read.  But I've learned by painful
> experience that to do so is unwise.

I do not have a big problem changing the code.  If someone has a Linux
kernel with garmin driver and a garmin device I'll do whatever it takes
to get it working.  I have no problem violating BCP and every standard
written if I have to do that to make the code work.  I have a problem
changing code that is not broken.  I have a problem changing code for a
target that can not even use or test the code!  I got enough real things
to fix.
