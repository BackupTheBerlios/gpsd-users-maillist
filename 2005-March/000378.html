Willem <willem at pcfish.ca>:
> One last question though - how come we have lost the ability to set the baud
> rate?  With this latest version my program has to wait for the server to
> hunt down the correct baud rate which takes some time.  Is it possible to
> put back the option to set the baud rate when starting gpsd?

Possible, but I'm reluctant to do it.  I want to *remove* options, not add
them -- I'd get rid of -f and -S if I could.  There are a couple of reasons for
this:

1. gpsd is intended to be zero-configuration, launched by a hotplug script 
and automatically syncing up with whatever GPSes it finds.  Adding a baud rate
option would represent a step away from that, back towards the bad old world 
where users have to do more than just plug the GPS into a handy port to make
it work.  

2. I may give gpsd the capability to handle multiple GPSes.  If that happens,
an -s option on the command line stops making a lot of sense.

Remember, my typical use case is Aunt Tillie with a GPS mouse and a laptop.
I'm trying to ruthlessly simplify all interfaces.  Considering time to first 
fix and the fact that you usually pay the hunt-time cost only once (when you 
start up your application) I don't think a second and a half of hunt time
is much of a price for the simplification.

Remco or one of the other senior developers might have a different opinion.
I could be argued into putting that option back, but I would rather spend
the effort on making autoconfiguration smoother and faster.

I'm going to try your recipe for reproducing your reconnect bug next.  If
I can tickle it, I'll fix it.
