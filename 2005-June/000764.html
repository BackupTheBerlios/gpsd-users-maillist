Ok, I've been using gpsd 1.10 for ages with kismet, and I decided to try
some new versions to see how it got better...

Quoting Eric S. Raymond who wrote on Thu, Feb 10, 2005 at 02:50:53PM -0500:

> Dragorn has a big rewrite of the relevant section of kismet in
> progress.  He's promised gpsd 2.0 support, but I don't know when it
> will ship.

So you know the results of tring gpsd 2.x versions with kismet.

gpsd 1.90 also has an interesting failure mode, where it gets confused by
the messages from my jupiter zodiac receiver[1] so it constantly hops between

M=1,P=Lon Lat

and

M=2

(2d fix, but no lon/lat)

So I'm back to 1.10 with a patch to keep it from coredumping on invalid
data.

One big difference I noted between 1.x and 2.x is that 2.x blocks when a
client connects before the gps is active. I found this a nice feature, that
I could start gpsd and clients before the gps became active. Now the gpsd
seems to block and makes the clients "hang" (not recieving any return data
from the tcp socket) too.

[1] the one from http://www.gpskit.nl/
