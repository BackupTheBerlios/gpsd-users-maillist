> On Tue, 9 Aug 2005, Belgabor wrote:
> 
> > Thinking about it, maybe it would be nice to have some time scripts with
> > gpsd? I'm thinking mostly of something that would benefit people that
> don't
> > want to run ntpd =)
> 
> Not sure what good you can do without ntpd.  Not any practical alternative
> to it that I know.
> 
> > It/they should mainly provide two functionalities:
> > a) Determination of fudge agianst (a) ntpd server(s)
> > b) Using this fudge in ntpd-less environments to set system time
> 
> The only way to know your time relative to an ntpd server is to run
> another
> ntpd server.  Unless you need the time to much better than a second
> there is no point to ntpd.  If you do not run ntpd then it is hard to
> keep your system to even one second.
> 
> The problem with keeping system time is harder than it looks.  You can not
> just jump the time forward and back every time you notice a difference
> between local time and true time.  If you jump back 2 seconds when
> your system is 2 seconds fast you may run cron jobs trice.  If you jump
> ahead 2 seconds then you may skip a cron job.  Make files depend on
> accurate
> time stamps to determine which files need to be rebuilt.
> 
> The only safe way to adjust the clock on a running system is the gently
> slow it down or speed up to until the system time is correct.  For that
> you need ntpd.
> 
> There had been discussion on having gpsd directly set the system clock
> before but that has not reached critical mass.  Any one that cares about
> time will just use ntpd and do it right.  The config file is trivial
> and it is installed on just about every Linux there is.
> 
> So, I suggest you just byte the bullet and jump in to ntpd.
