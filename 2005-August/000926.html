On Friday August 5, I'm flying to Michigan to spend the following week
at a martial-arts camp.  I'll be training in Western sword combat from
dawn to dusk.  For those of you curious about such things, go here:
<http://swordsmanship.net/index.php?page=News&article=2005+Sword+Camp>.

This necessarily means that my access to net will be intermittent or
nonexistent for about a week.  It's been nearly a month since 2.28.
Accordingly, I want to push out a release tonight or tomorrow, so
we can allow a few days for any dust to settle before I fly out.

I plan to ship with EverMore and iTrax support disabled, as these
drivers are still in development.  If the DARPA Grand Challenge guy
verifies that the merged code for their TrueNorth compass device is
good, I'll ship that.  The main new feature will in any case be
rtcmdecode.

Relative to 2.28, the repo code includes a couple of minor but
significant bug fixes for parsing of odd NMEA variants.  There
have been two architecture-level changes:

1) Maximum channels is now a device-type capability, not a #define.
   We can really support devices with more than 12 channels now.

2) A deep bug in the library code was fixed at revision 3002.  The
   result is that packets won't get lost when sent at intervals of
   more than one cycle time.  (This is the change that broke R=2
   operation for a day and a half until I found the loose end.)

If you have any last-minute bug fixes that haven't been merged, please
send them to me ASAP.
