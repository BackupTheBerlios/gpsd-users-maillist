> Hi,

> I've been following this discussion about PPS and NTP.

> From what I gather, the current system is a pure software mechanism?

I think there is some code to watch for control line changes...

> I know that some GPS's generate an actual pulse on a pin. It would
> seem to me that this would be the ultimate way of connecting NTP and
> the GPS, but of course there is any "standard" way of doing this.

We love it when hardware pps is available, but:

* not all receivers are designed to try to pps
* not all receivers have a pps signal out of the chipset
* not all receivers have the pps line tied to a serial line
* not all receivers tie the pps line to the same serial line
* not all receivers have a fixed relationship between the start of a
second and pps message
* usb introduces a huge amount of suck
- usb stacks do not necessarily operate in predicatable or consistent
time. not a huge amount, but it matters.
- usb introduces buffering delay. maybe not much, but it matters.
- usb serial emulation is truly serial, whereas rs232 is somewhat
parallel. control lines can be asserted independently of data lines
