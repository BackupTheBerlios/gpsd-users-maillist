Gary E. Miller <gem at rellim.com>:
> Because 32 bit UNIX time will fail at 03:14:08 UTC on January 19 2038.
> Any prgram not recompiled with whatever fix is used for that issue will
> fail at that time.
> 
> Once you hve the fix for 32-bit UNIX time you can start worrying about
> the year 2100 problem.

I know what the fix for the time_t problem is.  64-bit hardware --
which I'm sure will be nigh-universal in fifteen years, let alone
thirty.  Heck, I'm already on an AMD-64 box.  A 64-bit time_t won't
wrap around for approximately five hundred eighty five billion years.

Making a 21st-century assumption would only help gpsd in a rather unusual 
case.  The GPS has to be NMEA-only (e.g. not SiRF, not Garmin, not
Zodiac).  It has to not issue ZDA (Motorola and FV-18 chips issue ZDA).
*And* your system clock has to be so fubar that it's indicating the
wrong century.

Does this seem likely?
