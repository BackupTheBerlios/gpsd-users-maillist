Chris Kuethe wrote:
> On Thu, Aug 21, 2008 at 4:26 AM, Phil Endecott
> <spam_from_gpsd at chezphil.org> wrote:
>> Dear Experts,
>>
>> I've just bought myself an Ocean Server OS5000 digital compass:
>>
>>   http://www.ocean-server.com/compass.html
>>
>> It's a small board with a 3-axis accelerometer and a 3-axis magnetic
>> sensor, and it outputs heading, pitch and roll over a serial and/or USB
>> interface.  It has various configuration settings and at least one of
>> its output formats is NMEA sentences.
>
> Fun. I wish it had gyros too.

The manual makes a couple of references to gyros, so maybe there is 
another product like that in the pipeline.  There does seem to be some 
jitter in the heading that it reports and there's a configuration 
setting for smoothing; presumably a gyro would fix all that.

>> I note that gpsd claims to work with another digital compass from True
>> North Technologies; I have the impression that that's a similar device,
>> though rather more expensive.  So presumably it will or could be hacked
>> to work with my device.
>
> We only claim weak compatibility with the TrueNorth device. None of
> the developers have one and getting access to one seems to be
> difficult. There seem to be some quirks with device initialization...
>
>> What I want to do is to get both the compass data AND position
>> information from a GPS connected to another USB port.  But as I
>> understand it, gpsd does not attempt to merge data from more than one
>> source and in fact the application will get one or the other, randomly.
>
> close. gpsd won't merge data, but it will keep the streams separate.
> you can switch between which device you're querying with the F
> command.
>
>> Have I understood this correctly?  Is anyone using the True North
>> device in combination with a GPS?  Could gpsd be modified to do what I
>> want, or am I better off writing my own code?
>
> I think we had 1 TrueNorth user - haven't heard from him in ages.
>
> i'm not so keen on the current approach to overloading the meaning of
> various gpsd internals to map in devices other than GPSes. I think
> some refactoring needs to happen if we want to start being a
> generalize random-fragments-of-navigation-state relay daemon.
>
> If you're just looking to propagate NMEA data from this compass gpsd
> can probably relay it in "R=1" mode, though no processing will be done
> - it's just pass-through.

Thanks; that all sounds sensible.  It's easier for me to just parse the 
data from the compass separately, especially as it has some non-NMEA 
modes where the parsing is trivial.


Regards,  Phil.
