i'd like to reiterate that gpsd is meant to be portable software ...
"supports linux" is somewhat better than "designed for windows" but it
still shows a certain sort of tunnel vision. we want gpsd to work with
every OS or runtime that supports a good set of POSIX features. I
suppose you could run GPSD in a realtime singletasking environment if
your runtime library supported enough of the functions we use.

getting back on topic, this is why we try catalog receivers on the
hardware page by chip, interface and protocol. if your receiver speaks
one of our supported protocols ... great! it's compatible with gpsd.
if your receiver has some discernable attributes, we can let you know
about those in advance, but usually they don't matter. if your
receiver uses an interface that's supported by your OS, gpsd will
probably be quite happy with it. if your receiver has an unsupported
interface, then no application would be able to use it. there's almost
nothing that gpsd does that depends on the interface - bluetooth,
prolific, ftdi, silabs, ... doesn't matter so long as the kernel makes
it look like a tty.
