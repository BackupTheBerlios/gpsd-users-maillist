Hi everybody!

I've been having some problems setting up gpsd (tried versions 2.34 and
2.37) with a gateworks branded gps module (a trimble copernicus) on a
gateworks avila ixp4xx board running OpenWRT (kamikaze).
I've already jumped through some hoops to activate the serial port and get
the hardware set up, and now I'm able to see output with "cat /dev/ttyS0." (
when I run gpsd )

the trouble is, when I run "gpsd /dev/ttyS0 -N -D6" it scans through the
baud rates, and doesn't recognize the correct speed. (4800 8n1) gateworks
provides patches to change the initialization string of the generic nmea
driver to suposedly support the trimble, but gpsd can't connect to the gps,
regardless of whether the patches are applied or not. In the small window of
time when the autobaud is searching on 4800, I can see clear,
healthy-looking nmea coming out of /dev/ttyS0, complete with location data.
But connecting to the gpsd instance over telnet it sees no satellites. When
running in debug mode gpsd spits out packet sniffer errors before giving up
on 4800 and switching it's speed to 9600. It proceeds to iterate through all
the other potential baud rates before giving up and dying.

the strangest part is that *sometimes* it locks to the 4800, and although
gpsd still won't report position, the autobaud at least finds a fix.

I used the openwrt makefile as my starting point, and when it failed to work
with the default set of configure arguments, I deleted many of them (thus
enabling a whole bunch of stuff in hopes of magically fixing the problem. as
it stands, here are my configure vars. i'm linking against libstdcpp, linux
kernel is 2.6.25.

--without-x
--disable-pps

cheers,
