Hi hi--

In message <Pine.LNX.4.64.0609271334580.29291 at catbert.rellim.com> you write:

>I have heard rumours the TSIP has been broekn a while.  It would be
>good to know if the current code works.  It is iffy right now
>on the Garmin Serial binary.  There are severl protocols that
>use <DLE> delimiters and gpsd has trouble picking them apart.

I've tested with TSIP and it seemed to work, in that gpsd was
able to (sometimes slowly) recognise the TSIP code and not
coredump on me. Not really much more than a "pass/fail" test
actually, so don't place too much faith in my report.


I also ran gpsd with  NMEA/gps35 and it worked as well. In this
case I setup ntpd with the two shm clocks and gpsd did write the
timestamps to ntpd ok. 

One issue, is that gpsd seems to turn on a number of extra NMEA
strings. These extra strings impacted the linuxpps operation
as the plots show in the link:

http://www.wraith.sf.ca.us/ntp/linuxpps-gpsd.html

Note that linuxpps was not using gpsd at all, the linuxpps host
only saw the effect of the extra (other than just $GPGGA) NMEA
data strings.

This is really only a "beware of side effects" comment with
regards to gpsd, not a bug report.
