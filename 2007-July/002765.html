Eric S. Raymond wrote:
> Chris just posted a patch implementing an -l option to change the
> gpsd listener address
[...]
> I'm not happy with the idea of a
> command-line option,
[...]
> We could put this option in an optional /etc/gpsd.conf config file.  But then
> we'd have to write an interpreter for the config file syntax.  I dislike the
> thought of adding that much complexity overhead.
[...]
> So.  Now, let's suppose that we implement "listen-to" as a protocol
> command taking a listener-address argument.  Then, we add an option
> (say, -c) that reads protocol commands from standard input at startup time.
> 
> Now, to get the effect of a config file, we invoke gpsd as 
> "gpsd -c </etc/gpsd.conf" and we're done.

   Sounds good.  One minor suggestion would be to make the hypothetical
"-c" option take as argument the input file (with the usual "-" for
stdin).  This is to keep in line with the way most other applications
seem to work.  Effectively, one would have a configuration file which
uses protocol syntax, thus doing away with the need to implement a
different parser.


> Chris has made a strong privacy case that the default listener address
> ought to be 127.0.0.1, so that your client has to be running on the same
> machine that's hosting the GPS.  Those of you doing remote-telemetry things
> would have to set a more promiscuous listener address.

   As an aside, those of us doing remote telemetry at some point would
also probably like to protect the integrity and confidentiality of our
data by cryptographic means.  However, would I be mistaken in saying
that the usual existing tools (stunnel, ssh) fill in that role
adequately, saving us the need to implement this functionality into the
daemon itself?

Diego Berge.
