Yo Sebastian!

On Wed, 3 Feb 2010, Sebastian Schuberth wrote:

> However, according to [1], gpsd should
> spawn a thread, not a process, to wait on the PPS signal change.

gpsd uses fork() to daemonize (gpsd.c line 182) but then promptly
exits the foreground process.

gpsd uses pthread_create() to make the PPS thread (libgpsd_core.c line 396).

Grepping the code for fork() and pthread_create() show me no other
uses in gpsd.  So I am still confused as to your 3rd thread/process.
Maybe your forground gpsd process is hanging?  That can happen if
you fail to redirect stdin, stdout and stderr in some cases.

The difference in linux between thread and processes is so small as
to be almost irrelevant.  Only in bloated OS like WinBlows is the
distiction really significant.

> Or is
> busybox' ps listing threads as processes?

Dunno, don't really use busybox myself.  I use pstree mostly and it
treats threads and processes similarly.
