How will using /dev/cu.xxx, instead of /dev/tty.xxx, effect gpsd?  I 
see some undesirable behavior on Mac OS X, with Bluetooth & Bluetooth 
serial devices.

When an application (gpsd, Kismac, GPSUtility) connects to the 
Bluetooth serial device (I've been using /dev/tty.gps), the Bluetooth 
connection is made with the device, and everything works fine.  If the 
applications is then closed, or closes /dev/tty.gps, the Bluetooth 
connection remains active for 60 seconds.  If an application tries to 
use the Bluetooth device, /dev/tty.gps, while the Bluetooth connection 
is still active from the first open, the application just hangs, with 
NO data coming from the device.  From tracing gpsd, the application can 
open the device, it just doesn't receive anything from it.  If I 
manually disconnect the Bluetooth device (via System Preferences), or 
wait for the 60 second timeout, everything works fine.  This REALLY 
hurts with gpsd, when all clients disconnect, and then a client 
reconnects, within the 60 seconds.....

I should also mention, that some applications filter the list of 
devices you're allowed to use.  Kismac only allows the /dev/tty.xxx 
devices.  GPSUtility only allows /dev/cu.xxx devices.  dataComet (a 
GREAT & ACCURATE, terminal emulator), only allows /dev/cu.xxx devices 
for initiating a serial connection.  Both /dev/cu.xxx devices & 
/dev/tty.xxx devices are created when using either Bluetooth serial 
ports, or USB->Serial adapters.

I didn't know if my odd Bluetooth Serial behavior might be related to 
using /dev/tty.xxx, instead of /dev/cu.xxx.  Any thoughts?
