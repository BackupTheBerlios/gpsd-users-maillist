On 9/4/07, Marshall Crocker <crocker at cavs.msstate.edu> wrote:
> When it comes down to it, if you do manage to port gpsd to
> windows, someone is going to have to maintain this port. I
> don't know how willing the gpsd developers would be to take
> on this responsibility...

not even a little bit. that would mean i'd actually have to have a
windows box of some decent specification. the only thing that comes
close is a dual-boot partition on my laptop so i can look stuff up
with ms streets & trips[1], and occasionally run some vendor-supplied
gps tools[2]. for the last 10 years or so, i've been computing happily
without attachment to windows, and i don't intend to develop such an
attachment now.

[1] i bought  a tomtom, and use google maps a lot more, and gpsbabel
talks to both of those.
[2] i've written stuff in perl to do much of what sirfdemo does, so
it's certainly not beyond me to replicate this functionality on UN*X
if i really wanted to be an anti-windows zealot.

> so it may be a more effective use of your efforts to put together
> a windows gpsd cygwin package and direct users to download
> and install it whenever they install gpsdrive.  I would love to see
> a native windows version of gpsd but cygwin gets the job done
> and licensing isn't an issue between the two.

i have no objection to seeing a windows port. i won't feel hurt if i
never see any diffs either (the license does allow that sort of
behaviour), but some other users might be interested. if the diff is
clean, and doesn't mess up the rest of the source tree, i might even
consider integrating it. i'd be more inclined to look favourably on
windows portability code if someone stepped up and committed to
ongoing maintenance.
