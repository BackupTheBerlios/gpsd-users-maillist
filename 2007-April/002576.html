Matt Roberds <mattroberds at cox.net> writes:

> On Wed, 18 Apr 2007, Eric Pretorious wrote:
>
>> 1. ./autoconf.sh assumes that prefix=/.
>>   Almost all of the other software that I've built/installed lately
>> correctly assumed prefix=/usr/local.
>
> I tried running just './configure' with no options and it set up to
> install in /usr .  This seems to be pretty common among programs that
> use GNU Autoconf.

It is not common.  The default autoconf behavior, used by almost all
programs, is to use /usr/local as prefix if one is not given.

> I suspect that Autoconf predates the wide use of packaged binaries
> (RPMs, Debian packages, etc) and it was then reasonable to assume that
> if you built some software, you most likely wanted it to go into /usr .

I think Autoconf does predate packages, and maybe even Linux, but /usr
is where the operating system stuff is and stuff one compiles goes in
/usr/local or some other place.  This derives from BSD and SunOS.

> Now I suspect it's slightly more common for admins to let only the
> package manager put things in /usr , and to put stuff built from source
> into /usr/local .

I think it's a bad idea to put hand-compiled things in the same prefix
as packages.


The choice of /usr comes from

 AC_PREFIX_PROGRAM(gcc)

which has now been commented out of configure.ac.  This is apparently
from the original modification of gpsd to use autoconf, and IMHO it just
did not make any sense.   Importantly, it violates the Principle of
Least Astonishment, writing into a place where everyone knows it should
not.
