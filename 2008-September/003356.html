Hello,

I've recently purchased a Trimble Lassen IQ to use for a homebrew NTP server
project I'm working on.  Because I was having problems with using the receiver
straight up with `ntp` (which is beyond the scope of this mailing list), I've
opt'd to give `gpsd` a shot with the shm driver through `ntp`.

With `gpsd`, it appears to work beautifully with the Lassen IQ without any
hassles ; I can see that it picks up the TSIP binary and I can also tag out the
NMEA output lines too, depending on the port I'm using off my breakout board and
baud rate.

I can get `ntp` to use the SHM device while I have `gpsd` running, but after
about a good 5 to 10 minutes, I get an extreme amount of delay and jitter and it
never comes back down and in my `ntp` test configuration, it ends up defaulting
to the local hardware refclock instead.  

I don't have a lot of experience off-hand using the shared memory driver through
`ntp`, and this Lassen IQ module I'm still learning about, so is this
particularly normal or just not a reliable thing to use?  I just don't
understand the bottleneck during polling between this shared memory driver and
`gpsd` that would cause this much unreliability for the refclock.

I'll gladly take any opinions or suggestions, and I don't have any of the logs
or output on hand at the moment, but will post if need be.

Any comments are welcome.

-Adam
