Ed W <lists at wildgooses.com>:
> OK, I think I mis-described my suggestion.  Python in the build machine
> is no problem, but it would be useful to re-enable --disable-python with
> the intention of eliminating all python code during the install phase,
> ie it basically means that python isn't available on the *target*
> machine. Is this reasonable?

Actually, no.  We're not using autoconf any more, so there is no place 
for --disable-python to live.  The equivalent for scons would be some
kind of python=no option.  The trouble with this idea is that in order
to run "scons install" you have to have python to support scons.

> At least with the non-head version I am using there are several .py
> scripts installed at present during make install (I will filter them
> manually during my package creation phase, but having it in the gpsd
> build side would at least help focus contributors on keeping cleaner
> division between script and non script code?)

The 'focus' would be an illusion, though.
