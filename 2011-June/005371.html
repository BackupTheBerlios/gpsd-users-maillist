Aha, I start to understand why we are talking past each other.  Please
re-read your sentence above and I think you can see why I'm
misunderstanding you?  I (incorrectly) read what you wrote as "you can't
run scons on the source machine if the target machine doesn't have python"

Apologies, but I think you can also see why I'm misunderstanding you? (I
am British..)


> If all you want is a python=no option that disables Python
> installation on the *source* machine, I could do that. But if I do,
> occasionally people who want the full install are going to
> accidentally set it wrong, and I'm doubtful the very minor convenience
> for binary packagers is worth the gripes I'll get about that.

I think your implicit assumption is that there are only (4) binary
packagers in the whole world and everyone else is using their packages?
 If that were so then arguably we wouldn't even need a "make install"
script..?!

I think the point is that there are lots of folks who occasionally
install from source and aren't experts on the packages in question.  I
run a lot of packages from source and the ones where I need spend no
more than two mins reading the "./configure --help" output are by far
the most satisfying to work with.  I get that you aren't using
autotools, but I hope you will have it as a goal to give casual source
users a good experience where possible?

Why not make "python=yes" the default and then it should be really hard
for some binary packager to get this wrong? The point of "python=no" is
to facilitate less common uses.  I do think gpsd is one of those small
number of great packages which will be integrated into a very wide
variety of purposes in the long term (it's bordering on being a core
infrastructure component for some classes of use) - please try and keep
the flexibility that we can bend it to these uses without having to
fight the build system?
