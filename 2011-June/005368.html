On 29/06/2011 22:41, Eric Raymond wrote:
> Ed W <lists at wildgooses.com>:
>> OK, I think I mis-described my suggestion.  Python in the build machine
>> is no problem, but it would be useful to re-enable --disable-python with
>> the intention of eliminating all python code during the install phase,
>> ie it basically means that python isn't available on the *target*
>> machine. Is this reasonable?
> 
> Actually, no.  We're not using autoconf any more, so there is no place 
> for --disable-python to live.  The equivalent for scons would be some
> kind of python=no option.  The trouble with this idea is that in order
> to run "scons install" you have to have python to support scons.

We seem to be talking past each other again. "scons install" is fine
because this is run on the build machine - it's the *target* that will
be without python?

You are an incredibly experienced developer, so I hesitate to teach you
to suck eggs, but packaging is the process of grabbing the post
"install" phase to apply to some machine without the development
environment?

I haven't tested HEAD, but previous versions of gpsd have tried to
"install" .py files.  My suggestion was simply to add some flag to the
installer (don't care how) to not bother to do this.  For sure most
packaging systems have some method to suppress excess files, but it's
convenient if the build system does it for you?

Probably I'm missing the point, but forgive me for over stressing the
obvious?
