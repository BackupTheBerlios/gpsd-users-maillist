Hello,

don't know if this is a gpsd or a gpsdrive problem, seems like both, so
I send it to both mailinglists.

I use:
gpsd 2.28 (the new Debian package)
kismet 2005.06.R1
gpsdrive CVS version from yesterday

If I just plug-in my gps (Garmin GPS II in NMEA mode with a serial-USB
converter, reported by the kernel as pl2303) it will not work, for
instance xgps will hang during start-up.

I first have to do a killall gpsd, then stty -F .., which will not work
the first time, but will work the second time. In the kernel.log I now
will see: ttyUSB0: 1 input overrun(s) a number of times, usual 3 times.
After that I can do /etc/init.d/gpsd start and now xgps will work,
kismet will report the right position, but when I start gpsdrive it
might or might not report the right position.

Yesterday I drove around like that without gpsdrive reporting the
position (satellites according to gpsdrive 0/0), but gpsdrive got all
WLAN's found by kismet on the map in the right position, so I could
about see where I was by looking at the latest WLAN shown on the map.

If it does not work, I just will have to stop gpsdrive, stop kismet,
killall gpsd, stty -F .., start gpsd again, start kismet again, start
gpsdrive again, and now again it might or might not work. Once it works,
it seems to work stable.

One other thing I saw in the kernel.log which seems to relate to this
problem: usb 1-1: pl2303_write_bulk_callback - failed resubmitting write
urb, error -1

BTW, if I stop gpsd, then set the Garmin to Garmin mode, start gpsd
again and then start xgps, xgps will report: no data arriving. After
putting the Garmin in NMEA mode xgps will report the right position
again.
