Hi All,
I am running a pre-release version of familiar 0.8.4 (rc1) on an iPaq h2200 interfacing to a Holux GPSlim236 BT GPS via gpsd2.20.  I connect using the rfcomm layer and primarily use roadmap as an application (full details here: http://www.handhelds.org/moin/moin.cgi/HpIpaqH2200gps)  I have had to implement a somewhat heavy-handed fix for suspend/resume and I just wanted to see if this is a reasonable approach or if anyone has a better idea of where to deal with the issue.

Fundamentally gpsd works great - I start up the app, it makes a call to gpsd, gpsd binds to /dev/rfcomm0 and rfcomm handles the BT connection to the GPS (at least, this is my layman's understanding of it)

The problem arises when I suspend the ipaq (for example when I stop for groceries) - when I resume the ipaq, the program hangs with a GPSD,R=1 and does not update position.
It is quite easy to fix: I can either restart the program (which then makes a request of gpsd and....) or I can restart gpsd, and it reports out position fine.

I have added a script to the suspend/resume apmd scripts to kill gpsd on suspend and start it on resume.

Is there a better way to do this?  Is this fixed in a newer version?  I think it is most likely related to the first FAQ cited below.

I have read through the FAQ, specifically:
Why does the first A,E,O,P,T,U, or V command to a device always return "?"

Why do I have to restart gpsd whenever I power-cycle my Bluetooth device?


Any information on this would be greatly appreciated.
